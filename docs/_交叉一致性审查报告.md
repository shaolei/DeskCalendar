## 2. 严重缺陷（需修复后才可据此实现）

### 🔴 F1 — `80-Plugin/Event.md` 的 feature→plugin 编译依赖违反依赖倒置原则

**位置**：`80-Plugin/Event.md` §1 依赖方向（第 18–22 行）

```
internal/plugin  ◀─依赖──   internal/calendar     (emit DateChanged)
internal/plugin  ◀─依赖──   internal/ui           (emit PanelShown/Hidden)
internal/plugin  ◀─依赖──   internal/theme        (emit ThemeChanged)
internal/plugin  ◀─依赖──   internal/shell        (emit LifecycleChanged)
internal/plugin  ◀─依赖──   internal/app          (在 App 启动期装配总线)
```

**问题**：按 `Plugin.md` 第 20 行确立的记号（`A ◀─依赖── B` = "B 依赖 A"），上述写法表示 **`calendar`/`ui`/`theme`/`shell` 均编译依赖 `internal/plugin`**。这直接违反：

- `01-总体架构.md` §2 铁律："`plugin` 以接口钩子**反向依赖** feature/ui（依赖倒置，**不反向编译依赖**）"；
- 既定依赖方向 `app → shell → platform/state → feature → theme`，feature 不得依赖 plugin；
- `Plugin.md` 自身的依赖块（第 20–21 行）只列出 `app`/`shell` 依赖 `plugin`，**不含** `calendar`/`ui`/`theme`。

若按 `Event.md` 实现，calendar/ui/theme 将 `import "github.com/deskcalendar/deskcalendar/internal/plugin"`，造成**底层 feature 反向依赖插件层**，破坏整个分层与可测试性（feature 单测被迫引入插件包）。

**建议修复**：领域事件的**发射接口**不应放在 `plugin` 包内。应将事件发射 API 下沉到更低层（推荐 `internal/state`，或新增 `internal/core/events`），使：
- `calendar/ui/theme/shell` ──依赖──▶ `state`（emit 领域事件）；
- `plugin` ──依赖──▶ `state`（订阅/注册总线）。

并重写 `Event.md` 依赖块，删除所有 `internal/plugin ◀─依赖── internal/<feature>` 边，改为 `plugin → state` 与 `feature → state` 两条独立边。

## 3. 一般缺陷（建议修复）

### 🟡 F2 — `100-Release` 包映射三处不一致

| 来源 | 声明 |
|------|------|
| `01-总体架构.md` §4 表 | `100-Release ｜ release(build)` |
| `03-项目目录规范.md` §2 | **未列出** `release` / `build` / `updater` |
| `100-Release/Build.md` | 包 `build/`（仓库根，`github.com/deskcalendar/deskcalendar/build`） |
| `100-Release/AutoUpdate.md` | 包 `internal/updater` |
| `100-Release/Package.md` / `CI.md` | 子包 `build/packaging`、`build/nsis`、`build/target`、`build/assets` |

**问题**：
1. `01` 给出的包名 `release` 与模块文档实际使用的 `build` / `internal/updater` 三者互不匹配；
2. `03-项目目录规范.md` 的 `internal/` 映射**漏列** `build/`（根）与 `internal/updater`，导致"文档→包映射"权威表不完整；
3. `build/`（仓库根，非 `internal/`）与"所有代码包应在 `internal/` 下"的惯例不符（虽属构建期纯资源包，但应在 `03` 中明确例外）。

**建议修复**：在 `03-项目目录规范.md` §2 显式补充 `build/`（仓库根，零运行时逻辑）与 `internal/updater`（运行时更新器）；将 `01-总体架构.md` §4 行改为 `100-Release ｜ build(packaging/CI) + updater(运行时)`，消除 `release` 这一无对应实现的标签。

### 🟡 F3 — `40-Theme/Skin.md` import 路径缺 `github.com/` 前缀

**位置**：`40-Theme/Skin.md` 第 207 行

```go
"deskcalendar/internal/infra/fs" // 提供用户主题目录访问
```

**问题**：模块路径在 `02-开发规范.md` 定为 `github.com/deskcalendar/deskcalendar`。全仓其余 import 均写全称（如 `github.com/deskcalendar/deskcalendar/internal/infra/log`），唯独此处缺 `github.com/`，**无法编译**。

**建议修复**：改为 `github.com/deskcalendar/deskcalendar/internal/infra/fs`。

### 🟡 F4 — `gogpu.App.Run()` 与 `desktop.Run(gogpuApp, uiApp)` 术语分裂

**位置**：
- 用 `gogpu.App.Run()`：`01-总体架构.md`(§3 表/序列图)、`02-开发规范.md`(§3)、`_模板与写作规范.md`(ADR-02 行)、`30-State/Signal.md`(§6)
- 用 `desktop.Run(gogpuApp, uiApp)`：`_模板与写作规范.md`(gogpu API 事实)、`10-Shell/App.md`(§3/§9)

**问题**：gogpu 真实入口按模板 API 事实为 `desktop.Run(gogpuApp, uiApp)`（`gogpu.NewApp` 仅构造），`gogpu.App.Run()` 疑似概念性误称。两处表述并存会让实现者混淆主循环 API 名称。

**建议修复**：全仓统一为 `desktop.Run(gogpuApp, uiApp)`；若确为 `gogpu.App.Run()` 请在校验 POC 后统一回改模板与 `App.md`。

### 🟡 F5 — 渲染模式符号归属不明

**位置**：
- `20-Platform/WindowStyle.md` 第 130 行：在本包内定义本地 `RenderMode` 枚举（Auto/CPU/GPU）
- `10-Shell/App.md` 第 84 行：`gogpu.NewApp(Frameless, <渲染模式参数>)` —— 旧文档当作 gogpu 包符号引用（但 fork 并无该常量）

**问题**：同一个常量被两处分别"当作 gogpu 导出"和"在 platform 包内定义"，二者只能有一真，存在编译归属歧义。

**建议修复**：明确本地 gogpu fork 并无“宿主托管式”渲染模式常量，仅导出 `gogpu.RenderMode`（Auto/CPU/GPU）；`WindowStyle.md` 保留本地 `RenderMode` 枚举用于窗口样式簿记，弹窗在路径 D 下不引用 gogpu 渲染模式；统一 `gogpu.NewApp(gogpu.Frameless)` 写法。

### 🟡 F6 — `Plugin.md` 持久化声明与依赖块不一致

**位置**：`80-Plugin/Plugin.md` 第 163 行声称"插件启用状态持久化到本地 SQLite，复用 `internal/todo` 已引入的同一数据库文件"，但其 §1 依赖块（第 17–21 行）**未列出** `internal/todo`。

**问题**：若插件真的复用 todo 的 SQLite 句柄/文件，则 `plugin → todo` 的依赖应被声明（哪怕仅运行期共享文件）。当前依赖图遗漏该边，且"两个包各自打开同一 SQLite 文件"隐含 WAL/并发风险未说明。

**建议修复**：在 `Plugin.md` 依赖块显式补充 `internal/plugin ──依赖──▶ internal/todo`（共享 DB 文件/句柄），或在正文中说明插件通过 `internal/infra/fs` 自行管理独立 DB 文件（即与 todo 解耦）。

### 🟡 F7 — `gogpu.NewApp` 构造签名风格不一致

**位置**：
- `10-Shell/App.md` / `_模板`：`gogpu.NewApp(Frameless, ...)`（旧文档带 fork 不存在的渲染模式参数，符号未加 `gogpu.` 限定）
- `20-Platform/WindowStyle.md` 第 167 行注释：`gogpu.NewApp(gogpu.DefaultConfig().…)`

**问题**：同一 API 出现两种调用形态（位置参数 vs `DefaultConfig()` 链式），且 `Frameless` 是否带包限定前后不一。属代码样例可编译性瑕疵。

**建议修复**：在模板中固定一种 `gogpu.NewApp` 调用范式，全仓 10-Shell / 20-Platform 文档对齐。
