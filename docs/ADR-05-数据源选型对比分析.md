# ADR-05: 数据源选型（农历 / 天气 / 节假日）

**状态**：Accepted（已拍板 · 2026-07-07）
**日期**：2026-07-07
**作者**：Software Architect Agent
**关联**：ADR-01（gogpu/ui 框架）、ADR-02（gogpu/systray 托盘）、ADR-03（透明圆角）、ADR-06（Go 1.25+）、技术评估报告 v5 §10

---

## Context（背景与约束）

DeskCalendar 是**系统托盘时钟点击弹出的日历面板**（360 小清新日历复刻），月历网格每个日期需显示：农历/节气、节假日/调休标记；面板顶部可选显示天气（实况+短期预报，非 MVP）。

三类数据需求：

| 数据类型 | 用途 | 能否离线 |
|---------|------|---------|
| 农历 / 节气 / 干支 / 生肖 / 节日 | 月历网格逐日标注 | **必须离线**（算法可算） |
| 中国法定节假日 + 调休（补班日） | 月历"休/班"标记 | **应离线**（政府数据，可嵌入） |
| 天气（实况 + 预报） | 面板顶部模块（非 MVP） | **必须在线**（实时数据） |

**硬约束（来自已验证决策）**：
1. **零 CGO**：gogpu 全系纯 Go（v4 已验证 `CGO_ENABLED=0` 可编译运行），所有依赖必须纯 Go。
2. **离线优先**：常驻桌面组件，核心日历功能（日期/农历/节假日）断网也必须可用。
3. **Go 1.25+**：需对齐生态版本（ADR-06）。
4. **可逆性**：优先用接口隔离数据源，便于后续替换/升级，避免 early lock-in。

---

## 决策拆分

本 ADR 拆为三个子决策，彼此独立、已分别拍板：
- **ADR-05a**：农历 / 节气库 → ✅ `6tail/lunar-go`
- **ADR-05b**：天气数据源 → ✅ 非 MVP；默认 Open-Meteo，用户 key 切换和风
- **ADR-05c**：节假日 / 调休数据 → ✅ 每年拉取一次（holiday-cn）

---

## ADR-05a: 农历 / 节气库 ✅ 已定

### 候选方案

| 方案 | 纯Go | 零依赖 | 覆盖能力 | 维护状态 | 取舍 |
|------|------|--------|---------|---------|------|
| **A1. `6tail/lunar-go`** ✅选定 | ✅ | ✅ 无第三方依赖 | 公历/农历/节气/干支/生肖/星座/节日/宜忌/彭祖百忌 全覆盖 | 活跃（pkg.go.dev 2025-11 更新，v1.4.6） | 库体积偏大（多历法），但零运行时成本 |
| A2. `Lofanmi/chinese-calendar-golang` | ✅ | ✅ | 公历/农历/干支历转换（较精简） | 低活跃 | 缺节气细节/节日，需自补 |
| A3. 自实现天文算法 | ✅ | ✅ | 自定 | 自维护 | 工程量巨大；节气需天文计算，精度/边界易错 |

### 决策：采用 A1 `6tail/lunar-go`

**理由**：
- 唯一同时满足「纯 Go + 零依赖 + 零 CGO」且「功能全覆盖」的候选，直接命中月历网格所有标注需求（农历日期、节气、生肖年、传统节日）。
- 离线算法，无网络依赖，契合离线优先。
- 事实标准（Go 中文日历最常用），社区验证充分，降低 360 观感还原风险。

**我们放弃了什么**：
- 库代码量较大（多历法实现），vendor 体积略增——可接受，无运行时开销。

**确认项（已拍板）**：
- ✅ `lunar-go` 许可证为 **MIT**（Q4 用户确认），可商用/再分发，无阻碍。
- [ ] 构建时确认所需 Go 版本对齐 1.25 链路（v1.4.6 声明 `go 1.x`）。

## ADR-05b: 天气数据源 ✅ 已定（非 MVP）

### 候选方案

| 方案 | API Key | 费用 | 中国精度 | 特色能力 | 取舍 |
|------|---------|------|---------|---------|------|
| **B1. Open-Meteo** ✅默认 | **无需** | 完全免费 | 全球模型覆盖中国，够用但微观精度弱 | 实况/逐小时/7天/空气质量；免密钥零配置 | 缺中国特有：分钟级降水、灾害预警 |
| B2. 和风天气 QWeather | **需** | 免费版 2000次/日、10次/分；3天预报 | **最佳**（中国本土模型） | 灾害预警/分钟级降水/生活指数 | 需用户注册拿 key；key 管理与安全 |
| B3. 高德天气 | 需（复用地理key） | 免费额度 | 中国，较基础 | 若已用高德地理可复用 key | 天气数据丰富度弱于和风 |
| B4. 不接入天气（MVP 不做） | — | — | — | — | 缺失 360 核心模块，仅日历 |

### 决策：天气为**非 MVP 功能**；数据层接口隔离，默认 Open-Meteo，用户配置 key 切换和风

**架构原则**（Q1/Q2 已拍板）：
- 天气模块**不进入 MVP 范围**，先交付日历核心；但数据层 `WeatherProvider` 接口现在就定义好，避免日后返工。
- 默认实现 **Open-Meteo（零配置、免密钥）**。
- 当用户在配置中填入和风 API key 时，**自动切换**至 `QWeatherProvider`（升级路径，中国精度最佳），不改调用方。
- 天气始终**异步 + 缓存 + 限频 + 离线降级**（断网/无 key 时面板不显示天气模块，绝不阻塞日历主流程）。

```go
type WeatherProvider interface {
    Current(ctx context.Context, loc Location) (*Weather, error)
    Forecast(ctx context.Context, loc Location, days int) ([]*Weather, error)
}
// 配置驱动选择：Key 为空 → OpenMeteoProvider；Key 非空 → QWeatherProvider
```

**我们放弃了什么（选 B1 默认时）**：
- 中国微观天气精度与预警能力弱于和风——对"今日温度+短期预报"级需求足够；且用户可随时填 key 升级，决策可逆。

## ADR-05c: 节假日 / 调休数据 ✅ 已定

### 候选方案

| 方案 | 离线 | 调休准确 | 新鲜度 | 取舍 |
|------|------|---------|--------|------|
| **C1. 嵌入 `holiday-cn` JSON + 年度更新** ✅选定 | ✅ | ✅（抓国务院公告） | 年度手动/CI 更新 | 需每年更新一次数据文件 |
| C2. 运行时拉取 `holiday-cn`/`holiday-calendar` JSON + 嵌入兜底 | ✅（兜底） | ✅ | 自动最新 | 新鲜度依赖网络；需 fallback 逻辑 |
| C3. 自爬国务院公告 | — | ✅ | 最新 | 脆弱，站点改版即崩，维护成本高 |

### 决策：C1+C2 轻量混合 —— 构建期/CI 拉取 holiday-cn 烘焙进二进制（离线保证）+ 可选运行时年度刷新

**理由**（Q3/Q4 已拍板）：
- 调休是**政府行政决定，非算法可算**，必须靠数据。
- `NateScarlet/holiday-cn` 自动抓国务院公告、JSON 按年输出、含补班日，社区最可靠；许可证 **MIT**（Q4 确认），可再分发。
- 国务院**每年只更新一次**放假安排（Q3 确认），因此**每年拉取一次即足够**：在发布构建/CI 阶段拉取最新 JSON 烘焙进二进制，保证离线可用且新鲜；运行时再叠加**可选的一年一次远程刷新** + 嵌入兜底。

**我们放弃了什么**：
- 运行时实时性：仅年度刷新，无法日内更新——但国务院本就年度发布，无实际损失。
- 引入一次构建期网络拉取（CI 步骤）——成本极低，且有嵌入兜底不怕失败。

**落地形态**：
- `HolidayRepository`：内置嵌入 JSON（构建期烘焙，离线保证）为默认源；
- 预留远程覆盖接口：每年触发一次拉取 holiday-cn 最新数据更新本地缓存，失败则回退嵌入数据。

## 整体数据架构原则（汇总）

```
┌─ Shell / Feature 层 ──────────────────────────────┐
│  CalendarFeature (MVP)                              │
│    ├─ LunarService   → lunar-go (嵌入, 离线)        │
│    └─ HolidayService → HolidayRepository(嵌入JSON)  │
│                                                      │
│  WeatherService (非MVP, 接口就绪)                    │
│    └─ WeatherProvider(接口)                         │
│         ├─ OpenMeteoProvider  (默认, 免key)          │
│         └─ QWeatherProvider   (用户配key启用)        │
└──────────────────────────────────────────────────────┘
        │                │                  │
   纯算法/离线      嵌入数据/离线       网络(异步+缓存+降级)
```

1. **离线优先**：农历、节假日零网络依赖；天气为唯一在线源且必须优雅降级。
2. **接口隔离**：`LunarService`/`HolidayRepository`/`WeatherProvider` 均为接口，实现可替换、可测试、可 mock。
3. **零 CGO 合规**：所有候选均为纯 Go（lunar-go / holiday-cn 嵌入 JSON / Open-Meteo&和风 HTTP 调用），不破坏 ADR-01 生态约束。
4. **可逆**：天气 provider 由配置驱动热插拔；节假日可加远程刷新——决策不锁定。

## Consequences（采纳后的整体影响）

**变得更容易**：
- 核心日历功能完全离线，无外部服务可用性格点。
- 数据源被接口隔离，单测可用 mock，替换供应商零调用方改动。
- 零 CGO 链保持纯净，`CGO_ENABLED=0` 构建不受影响。
- 天气模块解耦于 MVP，可独立排期上线，不阻塞日历主流程。

**变得更难 / 需承担**：
- 农历库体积略增（一次性成本）。
- 天气走和风时需引入 key 配置与安全存储（仅当用户启用）。
- 节假日数据需年度维护（构建期烘焙 + 可选运行时刷新，成本极低）。
